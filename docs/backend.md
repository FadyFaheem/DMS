# Backend

> The Rails 8 API-only app: how it's laid out, the conventions that hold it together, and how to extend it.

[ŌåÉ Game Design](game-design.md) &middot; [Handbook](README.md) &middot; Next: [API Reference ŌåÆ](api-reference.md)

The backend lives in [`api/`](../api). It is **API-only** (`config.api_only = true` in [`config/application.rb`](../api/config/application.rb)) ŌĆö no views, no asset pipeline, no sessions. It speaks JSON and nothing else.

- **Ruby** 3.3 &middot; **Rails** 8.1 &middot; **PostgreSQL** via ActiveRecord &middot; **Puma** &middot; **RSpec**

## The core convention: thin controllers, fat services

Every controller does the same four things and nothing more:

1. Authenticate the player (inherited from [`BaseController`](../api/app/controllers/api/base_controller.rb)).
2. Run any required [compute-on-read](architecture.md#compute-on-read) tick.
3. Hand off to a **model** or **service** that holds the logic.
4. Render JSON via `GameSerializer`, or an `{ error }` with the right status.

`feed` is the whole pattern in one method:

```ruby
def feed
  dino = current_player.dinosaurs.alive.find(params[:id])
  Simulation::DinoTick.call(dino)          # advance time first
  Feeding.call(dino, diet: params[:diet])  # logic lives in the service
  render json: GameSerializer.dinosaur(dino.reload)
rescue Feeding::InsufficientFood
  render json: { error: "Not enough food of that type" }, status: :unprocessable_entity
rescue ArgumentError => e
  render json: { error: e.message }, status: :unprocessable_entity
end
```

Business rules ŌĆö costs, formulas, validation, side effects ŌĆö never live in a controller. They live in `app/services` (behaviour) and `app/models` (state + catalogs). If you find yourself writing an `if` about game rules in a controller, it belongs in a service.

### Error handling

Failures are returned as `{ "error": "human readable message" }` with an appropriate status ŌĆö almost always `422 Unprocessable Entity` for rule violations, `404` for missing records (via `rescue_from ActiveRecord::RecordNotFound`), and `401` for a bad/absent park code. Services raise typed errors (e.g. `Feeding::InsufficientFood`, `SpeciesAcquisition::Error`, `Prestige::Reset::Error`) that the controller maps to responses.

## Identity

There are no accounts and no passwords. A player is identified by a single **park code** ŌĆö a bearer token generated at signup.

[`PlayerAuthentication`](../api/app/controllers/concerns/player_authentication.rb) is a concern mixed into `Api::BaseController`. It runs a `before_action` on every API request:

```ruby
def require_player
  @current_player = Player.find_by(player_code: bearer_code)  # from "Authorization: Bearer <code>"
  return if @current_player
  render json: { error: "Invalid or missing player code" }, status: :unauthorized
end
```

So every controller has a scoped `current_player`, and all queries hang off it (`current_player.dinosaurs.find(...)`), which also enforces ownership for free.

The only unauthenticated endpoints are `POST /api/players` (create a park) and the two health checks. `PlayersController` opts out with `skip_before_action :require_player, only: :create`.

Park codes are generated by the [`PlayerCode`](../api/app/models/player_code.rb) model: four groups of four characters (`AB3K-7QPX-MN24-RT9F`) from an unambiguous alphabet (no `0/O/1/I/L`), regenerated until unique.

> This is intentionally low-security ŌĆö it's a casual game, not a bank. Don't add JWTs/passwords without a real requirement; see the "Do NOT" list in [`CLAUDE.md`](../CLAUDE.md).

## Models

Models come in two flavours.

### Persisted (ActiveRecord)

Backed by tables (see [Database](database.md)): [`Player`](../api/app/models/player.rb), [`Habitat`](../api/app/models/habitat.rb), [`Dinosaur`](../api/app/models/dinosaur.rb), [`Breeding`](../api/app/models/breeding.rb), [`Disease`](../api/app/models/disease.rb), [`Structure`](../api/app/models/structure.rb), [`Research`](../api/app/models/research.rb), [`FoodProduction`](../api/app/models/food_production.rb), [`Attraction`](../api/app/models/attraction.rb), [`ActiveEffect`](../api/app/models/active_effect.rb), [`Event`](../api/app/models/event.rb), [`SpeciesUnlock`](../api/app/models/species_unlock.rb), [`GoalCompletion`](../api/app/models/goal_completion.rb).

These hold state plus small, query-free helper methods ŌĆö e.g. `Player#income_multiplier`, `Dinosaur#status`, `Habitat#overcrowded?`.

### Static catalogs (plain Ruby modules)

Game *content* that never changes at runtime is defined in code, not the database, as frozen value objects:

[`Species`](../api/app/models/species.rb) &middot; [`TerrainCatalog`](../api/app/models/terrain_catalog.rb) &middot; [`ResearchCatalog`](../api/app/models/research_catalog.rb) &middot; [`FoodProductionCatalog`](../api/app/models/food_production_catalog.rb) &middot; [`StructureCatalog`](../api/app/models/structure_catalog.rb) &middot; [`AttractionCatalog`](../api/app/models/attraction_catalog.rb) &middot; [`DiseaseCatalog`](../api/app/models/disease_catalog.rb) &middot; [`EventEffectCatalog`](../api/app/models/event_effect_catalog.rb) &middot; [`GoalCatalog`](../api/app/models/goal_catalog.rb)

Each exposes `all` / `find(key)`. To add a species or tweak a research cost, you edit the catalog ŌĆö no migration. The per-player *state* that references them (which species you've unlocked, which research you own) is what lives in tables. The actual numbers in these catalogs are documented in [Game Design](game-design.md).

## Services

`app/services` is where the game actually happens. Services are plain Ruby (`module_function` modules or small classes) with a `.call` entry point, and they take an injectable clock/RNG so specs are deterministic.

| **Simulation** (compute-on-read) | [`ParkTick`](../api/app/services/simulation/park_tick.rb) (orchestrator), [`DinoTick`](../api/app/services/simulation/dino_tick.rb), [`Consumption`](../api/app/services/simulation/consumption.rb), [`FoodCollection`](../api/app/services/simulation/food_collection.rb), [`Events`](../api/app/services/simulation/events.rb), [`AttractionIncome`](../api/app/services/simulation/attraction_income.rb), [`HealthFormula`](../api/app/services/simulation/health_formula.rb) |

## Serialization

There is exactly one place JSON shapes are defined: [`GameSerializer`](../api/app/serializers/game_serializer.rb). Controllers call `GameSerializer.player(...)`, `.dinosaur(...)`, `.habitat(...)`, etc. ŌĆö never `render json: model` directly.

Two rules it enforces everywhere:

- **Datetimes are UTC ISO-8601** with a trailing `Z` (`time.utc.iso8601`), so the TypeScript client can parse them with `new Date(...)` unambiguously.
- **`GameSerializer.player` is the whole park** ŌĆö dinos, habitats, summary, research, species, farms, structures, attractions, active effects, goals, prestige, and recent events ŌĆö in one object. That single payload drives the entire frontend (see [API Reference ŌåÆ the player payload](api-reference.md#the-player-payload)).

## Routing

All game routes are namespaced under `/api` in [`config/routes.rb`](../api/config/routes.rb); the two health checks (`/health` and Rails' `/up`) sit at the root. The full surface is in the [API Reference](api-reference.md).

## Adding a resource end to end

Say you want `gift_cards`. The pattern, start to finish:

1. **Migration + model** ŌĆö `rails g migration CreateGiftCards ...`, `rails db:migrate`, add `app/models/gift_card.rb` with a `belongs_to :player`.
2. **Service** (if there's any logic) ŌĆö `app/services/gift_card_redemption.rb` with a `.call` that raises a typed error on failure.
3. **Serializer method** ŌĆö add `GameSerializer.gift_card(...)` (UTC datetimes!) and include it in `GameSerializer.player` if it's part of park state.
4. **Controller** ŌĆö `app/controllers/api/gift_cards_controller.rb < BaseController`, thin: tick if needed, call the service, render or rescue.
5. **Route** ŌĆö `resources :gift_cards` inside the `namespace :api`.
6. **Specs** ŌĆö a request spec under `spec/requests/api/` and a service spec under `spec/services/` (inject a fixed clock/RNG).
7. **Frontend** ŌĆö a typed module in `frontend/src/api/`; see [Frontend](frontend.md).

## Testing

Specs live in [`api/spec/`](../api/spec) (models, services, requests) and run with `bundle exec rspec`. Service specs inject a fixed clock and seeded RNG so the simulation is deterministic. See [Development ŌåÆ Testing](development.md#testing).
